# Contributing to Briefly

Thank you for your interest in contributing to Briefly! This guide outlines how we collaborate, whether you're a team member, partner, or interested contributor.

## Our Collaboration Principles

- **Excellence** — We maintain high standards in all work
- **Transparency** — Open communication about progress and challenges
- **Collaboration** — We work together toward shared goals
- **Continuous Improvement** — We learn from each engagement

## Getting Started

### For New Team Members

1. **Onboarding** — Your point of contact will guide you through initial setup
2. **Documentation** — Review relevant docs in the `/docs` folder
3. **First Project** — Start with a smaller engagement to understand our processes
4. **Mentoring** — Regular check-ins with experienced team members

### For Partners & Collaborators

1. **Scope Agreement** — Clearly define the engagement scope and deliverables
2. **Communication** — Establish communication cadence and channels
3. **Documentation** — Keep detailed records of work and decisions
4. **Review Process** — All work is reviewed before delivery

## Working on Projects

### Issue & Project Workflow

1. **Create an Issue** — Use appropriate issue template (Client Project, Assessment, Team Engagement, etc.)
2. **Add Context** — Include client name, scope, timeline, and success metrics
3. **Track Progress** — Update the issue with regular progress notes
4. **Document Deliverables** — Link completed work and documentation
5. **Close & Review** — Final check and client feedback before closing

### Branch & Commit Standards

#### Branch Naming

```
client-name/project-brief
assessment/company-name
engagement/team-training
docs/feature-name
```

#### Commit Messages

Keep commits clear and descriptive:

```
[CLIENT] Add architecture assessment for XYZ Corp
[DOCS] Update services documentation with new offering
[ENGAGEMENT] Document team training session outcomes
```

### Pull Request Process

1. **Use PR Template** — The template guides you through documentation
2. **Clear Description** — Explain what changed and why
3. **Link Related Issues** — Reference the engagement or project issue
4. **Request Review** — Get feedback before merging
5. **Verification** — Confirm changes are correct before merge

#### PR Review Checklist

Before requesting review, ensure:
- [ ] Documentation is clear and complete
- [ ] All related issues are linked
- [ ] Changes follow our style and standards
- [ ] No sensitive client information is exposed
- [ ] Relevant team members have context

## Documentation Standards

### Client-Facing Documentation

- **Clarity** — Use clear, professional language
- **Completeness** — Include all relevant context and recommendations
- **Formatting** — Use consistent headings, lists, and formatting
- **Confidentiality** — Avoid unnecessary internal details

### Internal Documentation

- **Detail** — Document decisions, lessons learned, and approaches
- **Context** — Explain the why, not just the what
- **Links** — Reference related documents and previous work
- **Currency** — Keep documentation up-to-date

### Engagement Documentation

For each client project, document:

1. **Engagement Overview** — Client, scope, duration, objectives
2. **Technical Assessment** — Key findings and recommendations
3. **Progress Notes** — Weekly or milestone updates
4. **Deliverables** — Code, reports, training materials
5. **Outcomes** — Results and metrics
6. **Lessons Learned** — What worked well, what could improve

## Code & Architecture Standards

### Code Review

When reviewing code or architecture:

1. **Functionality** — Does it solve the problem?
2. **Quality** — Is it maintainable and performant?
3. **Standards** — Does it follow best practices?
4. **Testing** — Are there appropriate tests?
5. **Documentation** — Is it well-documented?

### Architecture Reviews

For architecture decisions:

1. **Problem Statement** — What problem are we solving?
2. **Options Considered** — What alternatives were evaluated?
3. **Recommended Approach** — Which solution and why?
4. **Trade-offs** — What are the pros/cons?
5. **Implementation Plan** — How will we build this?

## Communication Guidelines

### With Clients

- **Responsiveness** — Reply within 24 hours
- **Professionalism** — Clear, professional communication
- **Proactive Updates** — Keep them informed of progress
- **Transparency** — Be honest about challenges and timelines
- **Meeting Notes** — Document discussions and decisions

### With the Team

- **Async First** — Use Slack for quick questions
- **Scheduled Syncs** — Weekly or bi-weekly syncs for complex discussions
- **Documentation** — Record decisions in the relevant issue or document
- **Collaboration** — Ask questions and share knowledge

### Meeting Standards

- **Agenda** — Share agenda 24 hours in advance
- **Notes** — Document key decisions and action items
- **Action Items** — Assign owners and due dates
- **Follow-up** — Share notes within 24 hours

## Quality Standards

### Pre-Delivery Checklist

Before delivering any work:

- [ ] Requirements met or clearly communicated if not
- [ ] Documentation is clear and complete
- [ ] Code/analysis reviewed for quality
- [ ] No critical issues or blockers
- [ ] Client expectations set appropriately
- [ ] Timeline is on track

### Post-Delivery Process

After completing work:

- [ ] Client feedback collected
- [ ] Results documented in the project issue
- [ ] Lessons learned captured
- [ ] Knowledge shared with team if relevant
- [ ] Case study potential identified
